# Agent 自驱动游戏模式

> 相关索引：[[README]]  
> 相关文档：[[tech-stack]]、[[game-design]]

## 设计动机

这是一个**开发工作流设计**，与游戏运行时内容无关。目的是让 AI 编程助手（如 GitHub Copilot）在开发过程中替代人工手动操作，通过命令行工具逐步推进游戏，用于验证游戏逻辑、探索地图行为、复现特定场景。

该模式的核心价值是：

- **替代手动测试**：开发者无需打开浏览器或手动构造 HTTP 请求，由 AI 助手直接调用 CLI 命令探索游戏逻辑。
- **完整的操作历史**：每次操作产生独立的时间戳存档，形成可回溯的状态序列，便于复现问题或对比行为变化。
- **架构解耦验证**：通过复用 `game.ts` 的纯逻辑层驱动游戏，验证业务逻辑与传输层的彻底分离。

---

## 架构分层

游戏逻辑层（`apps/server/src/game.ts`）是两种模式的公共底座，不含任何 HTTP 或 I/O 依赖。

C/S 模式通过 `app.ts` 在上层包了一层 Express 路由，状态以 `Map<sessionId, GameState>` 形式存活在服务进程内存中。

Agent 模式通过 `scripts/run-agent-game.ts` 直接调用逻辑层，状态持久化为 `apps/server/saves/` 目录下的 JSON 文件。两种模式共享代码，但**不共享运行时状态**。

---

## CLI 命令设计

Agent 与游戏的每次交互均为**单次进程、原子操作**，没有长驻进程或事件循环。每个命令独立运行，读取最新存档，执行动作，再写入新存档。

`start` 命令创建一张全新的地图，生成带时间戳的初始存档，并打印全量地图（所有格子显示为未揭开状态）供 agent 观察。

`status` 命令读取最新存档，输出当前地图（已揭开格子显示对应符号，未揭开显示 `?`）、玩家属性、最近日志以及所有未揭开坐标的完整列表，供 agent 规划下一步。

`reveal <x> <y>` 命令加载最新存档，对指定坐标执行揭开动作，将更新后的状态写入新的时间戳文件，并打印操作结果。

---

## 状态持久化策略

每次 `start` 或 `reveal` 都生成一个新文件，文件名格式为 `game-state-<ISO时间戳>.json`。目录不做任何清理，所有历史快照永久保留，形成一条完整的操作记录链。

`status` 和 `reveal` 命令通过文件名字典序排序，自动选取最新的快照作为当前状态。由于 ISO 时间戳天然具有字典顺序与时间顺序一致性，无需额外元数据索引。

`saves/` 目录已加入 `.gitignore`，存档不进入版本控制。

---

## 日志与输出分层

CLI 的输出分为两个层次，由不同工具负责：

游戏内容输出（地图、玩家状态、未揭开坐标列表）使用 `console.log` 直接打印，格式以人类和 AI 都易于解析为目标。

运维级日志（操作结果、写入路径、错误信息）通过 `pino` + `pino-pretty` 输出，带时间戳、级别和结构化字段，与游戏内容视觉上有明显区分。

## AI 助手的操作方式

当 AI 编程助手需要推进游戏时，工作流程如下：

首先运行 `status` 命令，读取当前地图状态。输出中包含：

- 格子符号矩阵（坐标对齐，揭开的格子显示类型符号，未揭开显示 `?`）
- 所有未揭开格子的 `(x, y)` 坐标枚举
- 玩家当前 HP、ATK、DEF、等级、经验
- 最近日志

根据上述信息，AI 助手选择一个目标坐标，调用 `reveal <x> <y>`，观察返回的格子类型和日志消息，再重复上述流程直到达成目标（如全部揭开、触发特定事件等）。

这一流程的本质与开发者手动测试完全相同，只是由 AI 助手代为执行命令。
